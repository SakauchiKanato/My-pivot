My Pivot
迷った数だけ、私は進んだ。五里霧中のロードマップを、確かな軌跡に変える。過去の選択・迷いを記録し、次の意思決定のときに「過去の自分」を参考人として参照できるアプリ。

これは何のプロジェクト？
ターゲット：重要な選択を繰り返す学生・院生 課題：「なぜその選択をしたか」の感情・文脈が記憶から消え、次の選択でゼロからやり直してしまう 価値：迷いをそのまま記録 → 自動でタグ整理 → タイムラインで振り返り、の1ループを提供する

技術構成
Python を主軸に、フロントだけ React を使う構成。
   領域 技術 言語     フロントエンド Vite + React TypeScript   バックエンド FastAPI Python   タグ整理ロジック ルールベース（将来LLM追加可） Python   DB SQLite + SQLModel Python   なぜこの構成かは 構成の選び方.md を参照。

セットアップ手順
1. バックエンド（Python）
cd backend

# 必要なライブラリをインストール
pip install fastapi "uvicorn[standard]" sqlmodel

# デモ用データを投入（初回のみ）
python seed.py

# サーバー起動
uvicorn app.main:app --reload起動後、http://localhost:8000/docs でAPIドキュメントを確認できる。
2. フロントエンド（React）
別のターミナルで：
cd frontend

# 必要なパッケージをインストール
npm install

# 開発サーバー起動
npm run dev起動後、http://localhost:5173 でアプリが開く。
:warning: バックエンドとフロントエンドの両方を起動しておく必要がある。

選択肢は3つ
構成① React + FastAPI（← 今回作った雛形はこれ）
[ブラウザ] ─ React(TS) ─ HTTP ─ FastAPI(Python) ─ SQLite
Python比率：バックエンド・ロジック・DBがすべてPython。フロントのみTypeScript
メリット：UIの自由度が高い（五里霧中モードの暗いUI・タイムライン可視化が思い通りに作れる）。AIをPythonで書ける
デメリット：2つの言語・2つのサーバーを管理する必要がある。フロント:left_right_arrow:バックの接続設定（CORS等）が必要
向いているチーム：Reactの経験を活かしつつ、ロジックをPythonで書きたい
構成② Streamlit（Python単一）
[ブラウザ] ─ Streamlit(Python) ─ SQLite
Python比率：ほぼ100%
メリット：単一言語で完結。セットアップが最速。管理するサーバーが1つ
デメリット：UIの自由度が低い。「暗い五里霧中モード」「凝ったタイムライン」の表現が困難。見た目で差別化しにくい
向いているチーム：とにかく速く作りたい／UIにこだわらない
ソース（Streamlitの用途と制約・公式）：https://docs.streamlit.io/
構成③ Next.js単一（TypeScript）
[ブラウザ] ─ Next.js(TS・フロント兼バック) ─ SQLite
Python比率：0%
メリット：1つの言語・1つのプロジェクトで完結。統合が最も楽
デメリット：Pythonを使えない。「基本Pythonベース」という希望に反する
向いているチーム：TypeScriptで統一したい

判断のポイント
チームで以下を話し合うと決めやすい：
1. 見た目の作り込みはどれくらい重要？

審査基準（山室先輩「楽しい・面白い」、石井先輩「体験へのこだわり」）を重視するなら → ①か③
機能が動けば見た目は二の次なら → ②でもよい
2. フロント:left_right_arrow:バックの接続を設定する時間はある？

①は2サーバー構成なので、接続（CORS）でつまづく可能性がある
そこに時間を使いたくないなら → ②か③
3. Pythonをどれくらい使いたい？

「基本Pythonベース」を最優先するなら → ②が最も忠実、①が次点
Pythonにこだわらないなら → ③も選択肢


おすすめの進め方
今回、構成①の雛形を実際に動く形で作ってあります。まずこれを触ってみて、

「2サーバーの管理が思ったより大変そう」と感じたら → ②に寄せる
「これなら作れそう」と感じたら → ①で進める
という判断ができます。雛形が動くことは確認済みなので、ゼロからではなく「動くものを見ながら」判断できます。

ここからは、私の仮説
3つの中では、構成①が「Python主軸」と「Reactの経験活用」と「デモの見た目」の3つを最もバランスよく満たすと考えます。ただし、これは「チームがフロント:left_right_arrow:バックのAPI接続でつまづかない」という前提に依存します。
もしDay 1でAPI接続に半日以上かかるようなら、構成②（Streamlit）に切り替えるのが安全策です。Streamlitは見た目の自由度を犠牲にする代わりに、接続の問題自体が発生しません。
この判断は実際にチームが手を動かしてみないと確定できないため、Day 1でまず構成①のAPI接続の疎通確認を最優先し、そこでの手応えで最終決定することを推奨します。断定はできません。
