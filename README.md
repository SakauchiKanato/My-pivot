0. このドキュメントの位置づけ
本ドキュメントは、これまでの議論で確定した以下の前提の上に立つ。

ターゲット：重要な選択を繰り返す学生・院生
価値提案：「過去の自分を、次の意思決定の参考人にする」
コア機能：五里霧中モード（入力）→ タグ整理 → ピボット・タイムライン（参照）
MVP範囲：入力 → タグ整理 → タイムライン表示 の1ループが動く状態
技術判断はすべて「3〜7日で審査員に見せられる状態にする」という制約から逆算している。

1. 技術選定と根拠
1.1 判断の大原則
ハッカソンにおける技術選定は「使いたい技術」ではなく「期間内に動くものを作れる技術」で決めるべきである。
チームがReact系経験を持つため、学習コストがゼロに近い技術に寄せることが最優先。新しい技術の学習に1日使うと、7日のうち14%を失う。
1.2 推奨スタック
   レイヤー 推奨技術 理由     フレームワーク Next.js (App Router) チームのReact経験を直接活かせる。フロント・バックを1つのコードベースで完結でき、期間内の統合コストが最小   言語 TypeScript 型により実装中のバグを早期発見。チーム開発で認識齟齬を減らす   スタイリング Tailwind CSS 「五里霧中モードの暗いUI」等のデザインを高速に実装できる   DB SQLite + Prisma（ローカル）または Supabase（クラウド） 下記1.3で詳述   状態管理 React標準（useState / useContext） ハッカソン規模ではRedux等は過剰。導入コストが利益を上回らない   デプロイ Vercel Next.jsとの親和性が最も高く、gitプッシュで自動デプロイ。デモ用URLが即座に用意できる   1.3 データベースの判断分岐
DB選定は、チームの状況で2択になる。
選択肢A：SQLite + Prisma（ローカル完結）

メリット：認証不要、セットアップが数分、オフラインで動く
デメリット：複数端末での同期不可、デプロイ時に工夫が必要
向く場合：デモを1台のPCで見せる、認証を作る時間がない
選択肢B：Supabase（クラウド）

メリット：認証・DB・ストレージが統合済み、写真添付の保存が容易
デメリット：セットアップにやや時間、無料枠の制約理解が必要
向く場合：複数端末デモ、写真機能を本格実装したい
3〜7日という期間では、まず選択肢A（SQLite）でMVPを完成させ、余裕があればBに移行する判断を推奨する。理由は、認証実装は「価値の核」ではないため、そこに時間を使うと審査で評価される機能実装が削られるため。

2. AI利用の判断（「まだ決めてない」への回答）
「タグ自動整理」機能でLLMを使うかは未確定とのことなので、判断材料を整理する。
2.1 段階的実装を推奨
結論：Day 1〜4はルールベース、余裕があればDay 5以降にLLMを追加する二段構え。
この判断は、過去の議論で繰り返し現れた設計原則（高信頼性のシンプルな機能を基盤とし、複雑な推論は限定的・明示的に加える）と一致する。
2.2 なぜルールベースを先に作るか
   観点 ルールベース LLM     実装時間 数時間 API連携＋プロンプト設計＋エラー処理で1〜2日   デモの安定性 100%予測可能 ネットワーク・API制限で失敗リスク   審査での説明 「確実に動く」 「AIを使った」加点はあるが動かないと減点   コスト 無料 API課金   ルールベースの具体例：
感情タグ「#絶望」「#つらい」「#どうしよう」 → カテゴリ「感情」
感情タグ「#進路」「#就活」「#院進」       → カテゴリ「意思決定」
マッピング表を辞書として持ち、部分一致で分類これだけで「タグが自動整理される」という体験は成立する。
2.3 LLMを足す場合の設計
もしDay 5以降に余裕があれば、「辞書に載っていないタグ」だけをLLMに渡す設計にする。全タグをLLMに投げるのではなく、フォールバックとしてのLLMにすることで、コストとリスクを最小化できる。
新規タグ入力
  → 辞書に一致？
     Yes → 即座にカテゴリ確定（無料・確実）
     No  → LLMに分類を問い合わせ（限定的利用）この設計は、審査で「AIを使っている」と説明でき、かつLLMが落ちてもアプリ全体は動き続ける。

4. データモデル設計
4.1 スキーマ（Prisma記法）
model Pivot {
  id          String   @id @default(cuid())
  title       String                        // タイトル（命名）
  content     String                        // 五里霧中の悩み（本文）
  layer       Layer    @default(PIVOT)       // 3層アーキテクチャ
  flag        Flag?                          // 成功/後悔（後から付与）
  confidence  Int?                           // 当時の確信度 1-5（下記4.3）
  imageUrl    String?                        // 添付画像
  createdAt   DateTime @default(now())       // 「いつ」検索軸
  tags        Tag[]                          // タグ（多対多）
}

model Tag {
  id        String   @id @default(cuid())
  name      String   @unique                 // 例：#絶望
  category  String                           // 変換後カテゴリ：感情
  pivots    Pivot[]
}

enum Layer {
  VISION   // ビジョン（長期目標）
  PIVOT    // ピボット（転換点）
  LIFE     // ライフ（日常）
}

enum Flag {
  SUCCESS  // 成功
  REGRET   // 後悔
}4.2 3軸検索との対応
過去の議論で定義した「いつ・意味・タグ」の3軸は、スキーマ上で以下に対応する。
   検索軸 対応フィールド     いつ（日付） createdAt   意味（題名） title   種類（タグ） tags → Tag.category   4.3 confidence フィールドについて
前回の議論で指摘したHindsight Bias（後知恵バイアス）問題への対応として、confidence（当時の確信度）フィールドを追加している。

flag（成功/後悔）は「今から見た評価」
confidence（1-5）は「決断した当時の確信度」
この2つの乖離を可視化できれば、「確信を持って選んだのに後悔した」「不安だったが成功した」といったパターンが見え、単なる成功/後悔の2値評価より深い自己理解につながる。
ソース（後知恵バイアスの原典）： Fischhoff, B. (1975). Hindsight is not equal to foresight. Journal of Experimental Psychology: Human Perception and Performance, 1(3), 288–299. https://doi.org/10.1037/0096-1523.1.3.288
ソース（デシジョン・ジャーナルにおける確信度記録の実践）： Farnam Street の決定日誌手法は、意思決定時に確信度（1-10）を記録し後から予測と現実を比較する。これにより意思決定バイアスを発見できるとされる。 https://fs.blog/

8. まとめ：この設計の核
この設計全体を貫く判断基準は一つ。
「価値の核となる1ループ（入力→整理→タイムライン）を、確実に動く状態で完成させる」ことを最優先し、それ以外はすべて『余裕があれば』に降格する。華やかな技術（LLM、写真、認証）はすべて加点要素であり、MVPの必須要素ではない。この優先順位が、限られた期間で審査に耐える成果物を生む。

ここからは、私の仮説
技術スタックの選定は上記で妥当だと考えるが、1点だけ検証が必要な仮説がある。**「Next.jsのApp Routerを全員が使いこなせるか」**である。App RouterはPages Routerより新しく、チームのReact経験が古いバージョン中心の場合、Server Components等の概念で詰まる可能性がある。
もしチームがApp Routerに不慣れなら、あえて枯れた Pages Router や、さらにシンプルな Vite + React（SPA）+ 軽量バックエンド に降ろす選択も合理的。この判断はチームの実際の習熟度に依存するため、断定はできない。Day 1の基盤構築で「詰まらないか」を早めに確認することを推奨する。


My Pivot
迷った数だけ、私は進んだ。五里霧中のロードマップを、確かな軌跡に変える。過去の選択・迷いを記録し、次の意思決定のときに「過去の自分」を参考人として参照できるアプリ。

これは何のプロジェクト？
ターゲット：重要な選択を繰り返す学生・院生 課題：「なぜその選択をしたか」の感情・文脈が記憶から消え、次の選択でゼロからやり直してしまう 価値：迷いをそのまま記録 → 自動でタグ整理 → タイムラインで振り返り、の1ループを提供する

選択肢は3つ
構成① React + FastAPI（← 今回作った雛形はこれ）
[ブラウザ] ─ React(TS) ─ HTTP ─ FastAPI(Python) ─ SQLite
Python比率：バックエンド・ロジック・DBがすべてPython。フロントのみTypeScript
メリット：UIの自由度が高い（五里霧中モードの暗いUI・タイムライン可視化が思い通りに作れる）。AIをPythonで書ける
デメリット：2つの言語・2つのサーバーを管理する必要がある。フロント:left_right_arrow:バックの接続設定（CORS等）が必要
向いているチーム：Reactの経験を活かしつつ、ロジックをPythonで書きたい
構成② Streamlit（Python単一）
[ブラウザ] ─ Streamlit(Python) ─ SQLite
Python比率：ほぼ100%
メリット：単一言語で完結。セットアップが最速。管理するサーバーが1つ
デメリット：UIの自由度が低い。「暗い五里霧中モード」「凝ったタイムライン」の表現が困難。見た目で差別化しにくい
向いているチーム：とにかく速く作りたい／UIにこだわらない
ソース（Streamlitの用途と制約・公式）：https://docs.streamlit.io/
構成③ Next.js単一（TypeScript）
[ブラウザ] ─ Next.js(TS・フロント兼バック) ─ SQLite
